Parsers ^OcaIhQKD

Se tendrá que conocer lo siguiente:
- Dirección de reconocimiento de la cadena de entrada
- Cómo se selecciona las reglas a la hora de realizar una transformación ^jvdSsDX2

Dirección de reconocimiento ^APRkAHsE

- Descendentes. Son aquellos que realizan el proceso a partir del símbolo inicial
                 e intentan llegar a la cadena de entrada.

- Ascendentes. Son aquellos que realizan el proceso a la inversa. ^NsQqZOb7

Selección de Reglas ^cDpnBpyK

- No-Deterministas (Backtracking)
        Se elige una de las reglas posibles y, si posteriormente no se llega a la
        cadena final, se retrocede a un punto en el que se hubiera podido elegir
        otra regla.

- Deterministas o Predictivos
        Cuando se presente la situación de tener que elegir entre varias reglas
        candidatas se asegura de coger la adecuada. Por ejemplo: ^OBqTRDaO

s ⟶ PRINT a ^DpWDWXku

a ⟶ b ^jRlrGE1C

a ⟶ c ^UTLjXgUx

b ⟶ IDENT NUM ^lXIzS5Ec

c ⟶ IDENT IDENT ^BeJyVRKQ

Supóngase la siguiente derivación: ^SCmS8Iud

s ⟹ PRINT a ^VY5SCHgn

Se podría ahora sustituir a tanto por la regla
 ^v9gZrUP8

a ⟶ b  como por a ⟶ c. Supóngase que se mira el
siguiente token y se encuentra un IDENT, por lo que
puede ser cualquiera de las dos. Buscamos el siguiente
token que vuelve a ser un IDENT, por lo tanto podemos
decir con seguridad que la regla a usar es la de a ⟶ c. ^2W7g9J4Y

        Clasificación de los parsers predictivos:
            - A un parser descendente se le denomina LL. Si además es predictivo se le
              denomina el parser LL(k). Siendo k el numero de tokens maximos que pueden
              predecir el parser.

            - De manera analoga se denomina LR a uno ascendente. Y LR(k) al ascendente
              predictivo. ^WjHncUlP

Símbolos directores ^JjiNmJdi

Los símbolos directores (SD) de una regla son aquellos tokens que, a la hora de  tener que elegir
entre varias reglas, si aparecen a la entrada indican que hay que seleccionar dicha regla. En realidad
son todos aquellos tokens que pueden aparecer como primer token en alguna de las 
cadenas que se pueden generar a partir de dicha regla.

Por ejemplo:
       
    a --> IDENT IDENT
          | b NUM

    b --> X | Y

Los símbolos directores de las dos reglas de a son:
    - SD(IDENT IDENT) es sólo el token IDENT
    - SD(b NUM) son los tokens {X, Y}

 ^5m06zNLG

Colisión de símbolos directores ^5d07NQgm

A la hora de elegir varias reglas con el mismo antecedente, no debería haber ningún
token que estuviera entre los símbolos directores de más de una de ellas.
Si esto ocurriera, al aparecer dicho token en la entrada, no se sabría cual 
de las reglas aplicar.

Lo que sucede con esto es que esta gramática no se podría implementar con un 
parser LL(1).

Para solucionar esto existen 3 soluciones principales:
    - Leer más tokens de la entrada (como hace ANTLR)
    - Cambiar de dirección de reconocimiento del parser
    - Buscar una gramática equivalente


 ^TfoOfVaG

Recursividad a Izquierda ^V2LvCmwk

La recursividad a la izquierda presenta un problema para los parsers LL(k). El problema
es que entraría en un bucle infinito:

            void a() {
                a();
                ...
            }

El ver una grámatica que use recursividad a la izquierda es una forma rápida de descartar
una gramática que pretenda ser implementada con un parser LL(k).

Para eliminar los problemas de la recursividad a la izquierda podemos usar el siguiente caso:

    


      expr -> expr ´+´ NUM
            | expr

 ^wybMNLb2

Ambigüedad ^vvoNtkJG

Un problema que se pueden encontrar en las gramáticas es que esta sea ambigüa, es decir, que 
la hay varias salidas para la misma entrada.

Para solucionar esto habrá que generar una gramática equivalente:

        expr ⟶ expr '+' expr
        expr ⟶ expr '*' expr
        expr ⟶ LITENT


Para ello usaremos la técnica de la cascada de precedencias, que consiste en realizar una separación
de los operadores por niveles en función de su prioridad y crear una lista por cada nivel. Quedando
la gramática anterior como:

        expr ⟶ expr '+' termino | termino           // lista 1+cs
        termino ⟶ termino '*' factor | factor       // lista 1+cs
        factor ⟶ LITENT


Ahora habrá que eliminar la recursividad a la izquierda:

        expr ⟶ termino otroTermino
        otroTermino ⟶ '+' termino otroTermino | ε
        
        termino ⟶ factor otroFactor
        otroFactor ⟶ '*' factor otroFactor | ε
        
        factor ⟶ LITENT
    
Esta ya seria implementable de una manera recursiva descendente.
Expresada en EBNF quedaria:

        expr ⟶ termino ('+' termino otroTermino)*
        termino ⟶ factor ('*' factor otroFactor)*
        factor ⟶ LITENT
 ^oMse6dPE

Ejemplo 410 ^VmkE9f6W

Se la siguiente gramática ambigüa:
       
        expr --> expr ´+´NUM
                | NUM
                | '(' tipo ')' expr


Podemos demotrar la ambiguedad con una entrada como por ejemplo:
    (int) 3 + 5

Ya que podemos entender la entrada como:
    - expr -> (tipo) | expr
    - expr -> exp | + | NUM

Hay esas dos ramas de lectura, por lo que se necesita cambiar la gramática.
Para solucionar esto hacemos lo siguiente:

        expr -> expr ´+´ NUM
               | NUM
               | '(' tipo ')' '<' expr '>'

Si limitamos asi la entrada se deja claro el alcance.

Otra posible solución es aplicar el orden de prioridad de ANTLR:

        expr -> NUM
                | '(' tipo ')' expr
                | expr  '+' NUM

Y con una gramática equivalente:

        expr -> termino
                | expr + termino
         
        termino -> factor
                   | '(' type ')' termino

        factor -> NUM

Árbol AST ^r3VeJAK3

El AST es el árbol mínimo que preserva la semántica de la entrada. Se caracteriza por no tener
los dos problemas del árbol concreto.

Para formar el árbol habrá que identificar los nodos del lenguaje. Se tendrán que representar
tanto las operaciones aritméticas como las relacionales. Unos ejemplos de esto son los siguientes:
asigna, while, lectura, programa, expr, var, lit

Los nodos pueden tener hijos por lo que habrá que identificarlos. Habrá que tener en cuenta lo siguiente:
    
    1. El otro tipo es nodo
    
    2. El tipo es una categoría sintáctica

    3. El tipo es un tipo propio del lenguaje

que es una lista. Por tanto
